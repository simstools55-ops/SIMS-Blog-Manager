# 改善ガイド

このページでは、改善ナビを見た後に何を直すかの考え方をまとめます。

## 改善ナビからSIMS-Coreへ渡す情報

Product 5.2.6では、改善ナビの依頼文を次の順番で作成します。

1. 記事基本情報
2. Search Console概要
3. Search Console上位クエリ（最大20件）
4. 改善目的
5. 改善優先順位
6. 記事ランク
7. 変更方針
8. 現在の記事本文データ（JSON）
9. 内部リンク候補
10. 内部リンク利用ルール
11. SIMS_FEEDBACK_V1の返却仕様

## Search Console上位クエリ

対象URLについて、クリック数・表示回数を優先して上位20クエリを依頼文へ追加します。各クエリには次の指標が付きます。

- クリック
- 表示回数
- CTR
- 平均順位

SIMS-Coreは、この実績を検索意図の判断に利用します。

## 改善優先順位

依頼文では、次の順番を基本とします。

1. SEOタイトル
2. 導入文
3. H2見出し
4. FAQ
5. 本文
6. 画像

既存本文の良い部分を残しながら、効果が出やすい箇所から改善するための目安です。

## 記事ランクと変更方針

記事ランクは次の形式でSIMS-Coreへ渡します。

- S（エース）
- A（安定）
- B（成長）
- C（育成）
- D（迷走）

また、広告コード、商品リンク、アフィリエイトリンクを変更しないことを依頼文へ明記します。


## 最新クエリの取得タイミング

改善ナビを起動した時点で、対象URLのクエリをSearch Consoleから毎回取得します。日次処理でクエリ詳細が保存されていない記事でも、改善時点の上位20クエリをSIMS-Coreへ渡せます。

画面には、取得成功・対象期間0件・取得失敗を区別して表示します。

## 内部リンク候補

改善ナビを開くと、記事DBとSearch Consoleの上位クエリから関連性の高いブログ内記事を3～8件抽出します。十分な関連性がない場合は、件数を満たすためだけに候補を追加しません。

候補には次の情報を表示し、SIMS-Core向け依頼文にも同じ内容を追加します。

- 記事タイトル
- URL
- 推奨アンカーテキスト
- 関連クエリ
- 関連度（★★★★★～★☆☆☆☆）

「関連理由」の文章やカニバリ候補は、今回の依頼文には含めません。

## 内部リンクの利用ルール

SIMS-Coreには次のルールを渡します。

- 候補は参考情報として扱う
- 本文の流れに自然に組み込める場合だけ採用する
- 無理に全候補を使わない
- 検索意図に合わない候補は採用しない
- アンカーテキストは本文に合わせて変更してよい
- テキストリンクはHTMLリンクを埋め込んだコピペ可能な完成形で出力する
- ブログカードが適切な場合は、挿入位置、URL、記事タイトル、採用理由を示す
- 各候補を採用・保留・不採用に分類する

## SIMS_FEEDBACK_V1

内部リンク候補を評価しただけでは、`changes.internal_links`は`false`です。記事へ実際に内部リンクを追加、置換、削除した場合だけ`true`になります。

### 最新クエリの確認

改善ナビを開くと、対象記事のSearch Consoleクエリをその時点で取得します。改善ナビ内には、取得件数、取得日時、対象期間、SIMS-Core向け依頼文へ使用する件数が表示されます。

「取得したクエリを見る」を開くと、上位クエリのクリック数、表示回数、CTR、平均順位を表で確認できます。取得処理中も、元の作業シートから画面が切り替わることはありません。


## メインクエリの確認

改善ナビの「Search Console最新クエリ」には、記事管理に登録されたメインクエリを表示します。取得した上位クエリに完全一致がある場合は印を付け、一致がない場合もその旨を表示します。
